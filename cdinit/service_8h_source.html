23/*
24 * This header defines service_record, a data record maintaining information about a service,
25 * and service_set, a set of interdependent service records. It also defines some associated
26 * types and exceptions.
27 *
28 * Service states
29 * --------------
30 * Services have both a current state and a desired state. The desired state can be
31 * either STARTED or STOPPED. The current state can also be STARTING or STOPPING.
32 * A service can be "pinned" in either the STARTED or STOPPED states to prevent it
33 * from leaving that state until it is unpinned.
34 *
35 * The total state is a combination of the two, current and desired:
36 * STOPPED/STOPPED : stopped and will remain stopped
37 * STOPPED/STARTED : stopped (pinned), must be unpinned to start
38 * STARTING/STARTED : starting, but not yet started. Dependencies may also be starting.
39 * STARTING/STOPPED : as above, but the service will be stopped again as soon as it has
40 * completed startup.
41 * STARTED/STARTED : running and will continue running.
42 * STARTED/STOPPED : started (pinned), must be unpinned to stop
43 * STOPPING/STOPPED : stopping and will stop. Dependents may be stopping.
44 * STOPPING/STARTED : as above, but the service will be re-started again once it stops.
45 *
46 * A scripted service is in the STARTING/STOPPING states during the script execution.
47 * A process service is in the STOPPING state when it has been signalled to stop, and is
48 * in the STARTING state when waiting for dependencies to start or for the exec() call in
49 * the forked child to complete and return a status.
50 *
51 * Interrupted transitions
52 * -----------------------
53 * A service that is STOPPING may be issued a start order, or a service that is STARTING may be
54 * issued a stop. In some cases a STOPPING or STARTING transition can be interrupted and immediately
55 * switch to the other kind of transition. We don't normally want to do this if we're waiting on an
56 * external process, since simply killing that process might leave a mess. However, if a service is
57 * waiting for its dependencies or dependents, its start or stop can usually be interrupted.
58 *
59 * Acquisition/release:
60 * ------------------
61 * Each service has a dependent-count ("required_by"). This starts at 0, adds 1 if the service has
62 * explicitly been started (i.e. "start_explicit" is true), and adds 1 for each dependent service
63 * which is not STOPPED/STOPPING (including dependents via a "soft" dependency relationship).
64 * When required_by transitions to 0, the service is stopped and it will release its own dependencies
65 * (unless it is pinned started). Conversely, when required_by transitions from 0 to 1, the service is
66 * started and dependencies will be required (unless pinned stopped).
67 *
68 * In general, therefore, the dependent-count determines the target state (STARTED if the count is greater
69 * than 0, otherwise STOPPED). Explicit activation counts effectively increments the count and sets the
70 * target state as STARTED.
71 *
72 * An exception is that setting the target state to STOPPED is used to inhibit restart. This is propagated
73 * to dependent services (i.e. their target state will also be set to STOPPED). The required_by count for a
74 * service in this scenario will settle to 0 anyway (because dependents will be stopped).
75 *
76 * Another exception is that a service may become STOPPED while it is still required (for example if a
77 * process terminates unexpectedly). This will force hard dependents to stop also, so again, the required_by
78 * count will settle to 0, unless the service is active and set to auto-restart (or a dependent is).
79 *
80 * When a service stops, any soft dependency links to its dependents must be broken, or otherwise the
81 * target state will remain as STARTED and the service will always restart. If the auto-restart option is
82 * enabled for the service, or if the service is explicitly being restarted (restart == true), this is
83 * actually the desired behaviour, and so in these cases the dependency links are not broken.
84 *
85 * Force stop
86 * ----------
87 * A service can be issued a stop-and-take down order (via `stop(true)'); this will first stop
88 * dependent services, which may restart and cancel the stop of the former service. However, a
89 * service can be force-stopped, which means that its stop process cannot be cancelled (though
90 * it may still be put in a desired state of STARTED, meaning it will start immediately upon
91 * stopping). Force-stop is achieved via a flag in the service record which is checked before
92 * interrupting a stop operation.
93 *
94 * Pinning
95 * -------
96 * A service may be "pinned" in either STARTED or STOPPED states (or even both). Once it
97 * reaches a pinned state, a service will not leave that state. A service that is pinned
98 * STOPPED cannot be started or marked active and any attempt to start it will fail. A service
99 * that is pinned STARTED remains started even if the service is not marked active and no other
100 * active service depends on it; this is the only scenario in which a service with a required_by
101 * count of 0 will remain started. Once unpinned the service will stop.
102 *
103 * (Note that pinning prevents, but never causes, state transition).
104 *
105 * The priority of the different state deciders is:
106 * - pins
107 * - force stop flag
108 * - desired state (which is manipulated by require/release operations)
109 *
110 * So a forced stop cannot occur until the service is not pinned started, for instance. (if
111 * pinned, the forced stop remains pending until the pin is released).
112 *
113 * Two-phase transition
114 * --------------------
115 * Transition between states occurs in two phases: propagation and execution. In both phases
116 * a linked-list queue is used to keep track of which services need processing; this avoids
117 * recursion (which would be of unknown depth and therefore liable to cause stack overflow).
118 *
119 * In the propagation phase, acquisition/release messages are processed, and desired state may be
120 * altered accordingly. Start and stop requests are also propagated in this phase. The state may
121 * be set to STARTING or STOPPING to reflect the desired state, but will never be set to STARTED
122 * or STOPPED (that happens in the execution phase).
123 *
124 * The two-phase transition is needed to avoid problem where a service that becomes STOPPED has
125 * an incorrect acquisition count, which may cause it to restart when it should not. The
126 * propagation phase allows the acquisition count to settle before the transition to the STOPPED
127 * state occurs, and the decision whether to restart can then be made based on the (correct)
128 * acquisition count. See "acquisition/release" above for details of when this can occur.
129 *
130 * Propagation variables:
131 * prop_acquire: the service has transitioned to an acquired state and must issue an acquire
132 * on its dependencies
133 * prop_release: the service has transitioned to a released state and must issue a release on
134 * its dependencies.
135 *
136 * prop_start: the service should start
137 * prop_stop: the service should stop
138 *
139 * Note that "prop_acquire"/"prop_release" form a pair which cannot both be set at the same time
140 * which is enforced via explicit checks. For "prop_start"/"prop_stop" this occurs implicitly.
141 *
142 * In the execution phase, actions are taken to achieve the desired state. Actual state may
143 * transition according to the current and desired states. Processes can be sent signals, etc
144 * in order to stop them.
145 */

957/*
958 * A service_set, as the name suggests, manages a set of services.
959 *
960 * Other than the ability to find services by name, the service set manages various queues.
961 * One is the queue for processes wishing to acquire the console. There is also a set of
962 * processes that want to start, and another set of those that want to stop. These latter
963 * two "queues" (not really queues since their order is not important) are used to prevent too
964 * much recursion and to prevent service states from "bouncing" too rapidly.
965 *
966 * A service that wishes to start or stop puts itself on the start/stop queue; a service that
967 * needs to propagate changes to dependent services or dependencies puts itself on the
968 * propagation queue. Any operation that potentially manipulates the queues must be followed
969 * by a "process queues" order (processQueues() method).
970 *
971 * Note that processQueues always repeatedly processes both queues until they are empty. The
972 * process is finite because starting a service can never cause services to stop, unless they

1085 std::unique_ptr<placeholder_service> ph { new placeholder_service(this, svc->get_name()) };
1086 add_service(ph.get());
